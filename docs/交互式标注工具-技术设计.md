# 交互式标注工具技术设计（F1）

| 项 | 内容 |
|---|---|
| 所属子系统 | 建库程序（数据录入前端） |
| 对应需求 | [需求分析.md](需求分析.md) F1 |
| 决策依据 | 半自动校正 + 爬虫车型名预填 |
| 依赖现有代码 | [detect_vehicle.py](detect_vehicle.py) 车辆检测、HSV 尾灯配对、亮灭判定 |
| 依赖待开发 | [朝向识别-技术设计.md](朝向识别-技术设计.md) `detect_orientation` |

---

## 1. 模块定位

交互式标注工具是建库程序的**数据录入前端**，也是整个项目的**数据瓶颈解锁器**：
- 半自动校正：程序自动跑检测流水线，人工只校正框、确认朝向/亮灭、校对车型标签。
- 产出标注数据，既用于**建库**（提取特征入库），也用于**训练**朝向识别/特征提取网络。

> 没有标注数据，朝向识别与特征提取网络都无法训练。本工具是后续所有训练模块的前置依赖。

---

## 2. 需求与约束

| 项 | 决策 |
|---|---|
| 标注形态 | **半自动校正**（程序自动检测，人工校正框 + 填标签） |
| 车型标签来源 | **AutoInfoScraper 爬虫车型名预填**，人工校正 |
| 输入 | 整车图片目录（含多角度，车尾视图优先） |
| 多车同图 | 支持逐辆分别标注 |
| 输出 | 标注 JSON（每图一条） + 直接送入建库流程 |

---

## 3. 总体流程

```
图片目录
  │
  ▼
[1] 元数据加载 ──▶ 从 AutoInfoScraper 抓取结果取每图的车型名
  │
  ▼
[2] 自动检测流水线（预填）
  │   车辆检测(detect_vehicle_full) ──▶ 朝向(detect_orientation)
  │   ──▶ 尾灯定位(detect_taillights 部件检测器) ──▶ 亮灭判定
  │   ──▶ 每车预填: vehicle_box / view / taillight_boxes / lit_status
  ▼
[3] 车型标签映射 ──▶ 爬虫名 ──▶ 标准车型系列（别名表）
  │
  ▼
[4] 标注界面（半自动校正）
  │   - 校正车辆框 / 尾灯框（拖拽）
  │   - 确认朝向(正/反/侧)
  │   - 确认亮灭状态
  │   - 校对车型标签（已预填）
  ▼
[5] 保存标注 JSON ──▶ [6] 送入建库流程提取特征入库
```

---

## 4. 自动检测流水线（预填）

复用现有 + 待开发的检测能力，对每张图自动产出预填标注：

| 步骤 | 实现 | 预填字段 |
|---|---|---|
| 车辆检测 | [detect_vehicle_full](detect_vehicle.py#L118) | vehicle_box, label(car/bus/truck) |
| 朝向识别 | detect_orientation（见朝向技术设计） | view (front/rear/side) |
| 尾灯定位 | [detect_taillights](detect_vehicle.py)（部件检测器 parts_v2） | taillight_boxes (L/R) |
| 亮灭判定 | [_judge_lit](detect_vehicle.py) | lit_status (bool) |
| 车型预填 | AutoInfoScraper 抓取名 + 别名映射 | model_series (标准名) |

> 流水线失败时（如未检测到车、朝向不确定），相应字段留空，由人工在界面补。

## 5. 标注界面设计

### 5.1 方案选型

| 方案 | 交互能力 | 开发量 | 依赖 | 评价 |
|---|---|---|---|---|
| OpenCV + 鼠标/键盘 | 基本框编辑、键盘选择 | 小 | 仅 cv2 | **★ 首期推荐**，快速验证 |
| PyQt5 / Tkinter | 完整 GUI（下拉/输入框/拖拽） | 中 | PyQt | 功能全，后续可升级 |
| Label Studio / CVAT | 专业标注平台 | 低（部署） | 服务化 | 偏离本工具集成目标，不推荐 |

**推荐 OpenCV 起步**：研究项目优先跑通流程，半自动预填已承担大部分标注工作量，界面只需做校正与确认，OpenCV 足够；后续若交互需求增长再升 PyQt。

### 5.2 界面元素

```
┌──────────────────────────────────────────┐
│  [图像显示区]                              │
│  叠加: 车辆框(绿) / 尾灯框(红) / 朝向标注   │
│  当前车辆 #2/3  朝向: rear  亮灭: LIT      │
├──────────────────────────────────────────┤
│ 车型: [丰田卡罗拉________] (预填，可改)     │
│ 朝向: [1]正 [2]反 [3]侧   (当前: 反)       │
│ 亮灭: [L]亮 [O]灭         (当前: 亮)       │
├──────────────────────────────────────────┤
│ [A]上一图 [D]下一图 [S]保存 [X]标记无效     │
│ [N]新增尾灯框 [Del]删除选中框 [Q]退出        │
└──────────────────────────────────────────┘
```

### 5.3 交互快捷键

| 键 | 动作 |
|---|---|
| A / D | 上一图 / 下一图 |
| 1/2/3 | 设朝向为 正/反/侧 |
| L / O | 设亮灭为 亮/灭 |
| S | 保存当前图标注 |
| X | 标记当前图为无效（跳过入库） |
| N | 新增尾灯框（鼠标拖拽） |
| Del | 删除选中框 |
| Q | 退出（自动保存进度） |

## 6. 车型标签映射（爬虫名 → 标准车型系列）

AutoInfoScraper 抓取的车型名通常含年款/配置后缀，需映射到标准车型系列：

| 爬虫抓取名（原始） | 标准车型系列 |
|---|---|
| 丰田卡罗拉 2021款 1.2T CVT | 卡罗拉 |
| 本田雅阁 260TURBO | 雅阁 |
| 大众朗逸 1.5L 自动 | 朗逸 |

### 映射机制
- **别名字典**（`model_aliases.json`）：`{"卡罗拉": ["corolla","卡罗拉","花冠"], ...}`，维护品牌→车型系列→别名列表。
- **映射流程**：爬虫名 → 去年后缀（正则去 `年款/排量/配置`）→ 别名字典匹配 → 命中则预填标准名，未命中则保留原始名并标记 `need_review`，人工在界面首次确认后自动加入字典。

## 7. 数据模型（标注 JSON）

每张图一条记录，存于 `data/annotations/{stem}.json`：

```json
{
  "image_path": "data/raw/corolla/001.jpg",
  "valid": true,
  "vehicles": [
    {
      "vehicle_box": [120, 80, 640, 420],
      "view": "rear",
      "lit_status": true,
      "taillight_boxes": [
        {"side": "left",  "box": [140, 300, 220, 360]},
        {"side": "right", "box": [540, 300, 620, 360]}
      ],
      "model_series": "卡罗拉",
      "model_source": "scraper_mapped",
      "need_review": false
    }
  ]
}
```

| 字段 | 说明 |
|---|---|
| valid | 是否有效图（无效图跳过入库） |
| vehicle_box | 车辆框 |
| view | 朝向（front/rear/side） |
| lit_status | 亮灭 |
| taillight_boxes | 尾灯框列表（含左右侧） |
| model_series | 标准车型系列名 |
| model_source | 标签来源（scraper_mapped / manual） |
| need_review | 映射未命中，需人工复核 |

## 8. 与建库流程衔接

标注保存后，调用建库提取流程（F7/F8）：

```
标注JSON ──▶ 按车辆读取 vehicle_box/view/taillight_boxes
            │
            ├─ view=rear  ──▶ 裁尾灯 ──▶ 特征提取 ──▶ 入尾灯库(带 lit_status)
            └─ view=正/侧 ──▶ 裁整车 ──▶ 特征提取 ──▶ 入整车库(兜底)
```

- 尾灯特征向量写入时附带 `lit_status`，供检索时状态过滤（见 [需求分析.md §7.3](需求分析.md)）。
- `need_review=true` 的记录暂不入库，待人工复核车型后补入。

## 9. 非功能需求

| 项 | 目标 |
|---|---|
| 自动检测预填 | < 2 秒 / 图 |
| 界面响应 | 实时拖拽无卡顿 |
| 持久化 | 每图标注即时落盘，防丢失；支持断点续标 |
| 批量 | 支持目录批量导入、进度记录（已标注/未标注） |

---

## 10. 落地步骤

1. **自动检测流水线整合**：串联现有车辆/尾灯/亮灭 + 朝向模块，输出预填标注。
2. **车型标签映射**：建别名字典 + 爬虫名解析，实现预填。
3. **OpenCV 标注界面**：图像叠加显示 + 框编辑 + 键盘交互 + 即时保存。
4. **建库衔接**：标注 JSON → 特征提取入库。
5. **标注首批数据**：产出朝向识别训练集（顺带为特征库积累样本）。

---

## 11. 风险与对策

| 风险 | 对策 |
|---|---|
| 爬虫车型名不规范、映射遗漏 | 别名字典 + need_review 机制，人工首次确认后自动入字典 |
| OpenCV 界面交互能力有限 | 首期聚焦校正/确认，预填承担主要工作量；后续按需升 PyQt |
| 大量标注耗时 | 半自动预填 + 快捷键 + 断点续标，降低人工成本 |
| 预填检测不准导致校正量大 | 随朝向/检测模块迭代，预填质量提升，校正量递减 |
