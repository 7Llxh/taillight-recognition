## 1. 项目背景与目标

### 1.1 背景
不同品牌/车型的尾灯在**形状、轮廓、颜色**上具有差异化特征。尤其在夜间亮灯时，尾灯特征清晰、无前大灯干扰，是判别车型的可靠依据。本项目通过构建尾灯的神经网络特征向量库，以"图像检索"的方式判别车型。

### 1.2 项目目标
构建两个相互配合的程序：
1. **建库程序**：输入图像，自动完成「车辆检测 → 朝向判断 → 部位识别 → 尾灯定位 → 特征提取 → 入库」，形成可检索的特征向量库。
2. **识别程序**：输入图像，走同样的前置流程后，通过「特征检索匹配」判别车型，并输出可视化结果。

### 1.3 总体流程

```
输入图像 ──▶ 车辆外轮廓检测 ──▶ 朝向识别(正/反/侧) ──▶ 部件识别
                                                              │
                       ┌──────────────────────────────────────┴──────┐
                       ▼                                               ▼
              朝向=车尾(反面)                                  朝向=正面/侧面
                       │                                               │
              尾灯定位+裁剪                                  整车轮廓/部件特征
                       │                                               │
              尾灯特征提取                                整车特征提取（兜底）
                       │                                               │
              检索【尾灯特征库】                          检索【整车特征库】
                       │                                               │
                       └─────────────────┬─────────────────────────────┘
                                         ▼
                              结果融合 + 置信度计算
                                         ▼
                            输出车型 + Top-K 候选 + 可视化
```

## 2. 范围界定

### 2.1 关键决策点（已澄清）

| 决策项 | 选择 | 影响 |
|---|---|---|
| 判别粒度 | **车型系列**（如「卡罗拉」「雅阁」，不细分年款） | 特征库以车型系列为基本检索单元 |
| 朝向逻辑 | **整车特征兜底** | 需双特征库：尾灯库（主）+ 整车库（兜底） |
| 部位识别 | **部件级识别**（尾灯/前灯/后视镜/车窗/轮胎/车牌/格栅等） | 辅助朝向判断+尾灯定位，并作独立能力；需部件标注+训练 |
| 输入形态 | **静态图片为主** | 视频流识别列为后续扩展，不在本期范围 |
| 标签录入方式 | **交互式标注（半自动校正）** | 程序自动检测，人工校正框 + 填车型标签；建库前端需实现标注工具 |
| 朝向粒度 | **正/反/侧三类** | 首期不区分左右侧与斜角，F3 按 3 类设计（见 [朝向识别-技术设计.md](朝向识别-技术设计.md)） |
| 亮/灭灯库组织 | **统一库 + 状态过滤** | 尾灯库不分亮灭子库，检索时按 lit_status 过滤 |
| 特征维度 | **512 维** | ResNet 常用维度，精度与存储平衡 |
| 标签来源 | **爬虫车型名预填** | AutoInfoScraper 抓取车型名预填，人工校正 |
| 多车同图 | **支持各自判别** | 复用多车检测，逐辆独立判别（零额外成本） |

### 2.2 纳入范围（In Scope）
- 单张静态图片输入的建库与识别流程
- 车辆外轮廓检测、朝向识别、部件识别
- 尾灯定位、配对、亮/灭灯判定、归一化裁剪
- 尾灯特征提取与检索（主路径）
- 整车特征兜底检索（正/侧视图）
- 特征库的构建、索引与维护
- 结果可视化输出

### 2.3 排除范围（Out of Scope / 后续扩展）
- 视频流识别与跨帧跟踪（规划 Phase 4，本期不做）
- 部件级语义分割（车轮/后视镜/车牌等精细分割）
- 具体年款级判别（如「卡罗拉 2021 款」）
- Web 服务 / 移动端部署（本期为本地脚本）
- 实时性要求（不面向在线高并发场景）

## 3. 用户与使用场景

### 3.1 目标用户
- 研究者/学习者本人：验证「以尾灯特征检索判别车型」的方法可行性。

### 3.2 典型场景
1. **建库**：将一批已知车型的车辆图片（含车尾视图）导入建库程序，程序自动提取特征并入库。
2. **识别**：输入一张未知车型的图片，程序输出判别结果（车型系列 + 置信度 + Top-K 候选）。
3. **多车同图**：一张图中含多辆车时，分别检测并各自判别。

## 4. 系统总体架构

### 4.1 两大子系统

```
┌─────────────────────────┐         ┌─────────────────────────┐
│      建库程序            │         │      识别程序            │
│  (Feature Library        │         │  (Recognition &          │
│   Builder)               │         │   Retrieval)             │
├─────────────────────────┤         ├─────────────────────────┤
│ 图像+标签导入             │         │ 图像导入                  │
│ 车辆检测 / 朝向 / 部位    │         │ 车辆检测 / 朝向 / 部位    │
│ 尾灯定位 / 特征提取      │         │ 尾灯定位 / 特征提取      │
│ 整车特征提取              │         │ 整车特征提取              │
│ 入库（尾灯库+整车库）    │         │ 检索匹配 + 结果融合       │
│ 库维护                    │         │ 可视化输出               │
└────────────┬────────────┘         └────────────┬────────────┘
             │                                    │
             ▼                                    ▼
      ┌──────────────────────────────────────────────┐
      │            特征向量库（FAISS 索引）            │
      │  ┌──────────────┐      ┌──────────────────┐   │
      │  │ 尾灯特征库    │      │ 整车特征库(兜底)  │   │
      │  │ (主路径)      │      │ (正/侧路径)      │   │
      │  └──────────────┘      └──────────────────┘   │
      │            + 元数据库（车型标签、朝向、样本路径）│
      └──────────────────────────────────────────────┘
```

> **关键架构点**：因选择「整车特征兜底」，系统需维护两个特征库。车尾视图走尾灯库检索（精度高）；正面/侧面视图走整车库检索（兜底）。识别程序按朝向分流，并支持结果融合与置信度回退。

## 5. 功能需求详述

> 编号规则：**F** = 功能需求，**P** = 公共能力。每条需求后给出验收要点。

### 5.1 建库程序（Subsystem A）

**F1 — 图像与标签导入（交互式标注，半自动校正）**
- 录入方式：**交互式标注，半自动校正**。
- 流程：程序自动跑检测流程（车辆检测 -> 朝向识别 -> 尾灯定位 -> 亮/灭判定），人工在界面上校正不准的框、填入车型系列标签，确认后入库。
- 标签来源：车型系列标签由 AutoInfoScraper 抓取的车型名预填，人工校正（详见 [交互式标注工具-技术设计.md](交互式标注工具-技术设计.md)）。
- 输入：整车图片目录（含多角度，车尾视图优先）。
- 支持单图多车时分别标注。
- *验收*：能批量导入图片、展示自动检测结果、人工校正并录入车型标签后入库。

**F2 — 车辆检测与外轮廓提取**
- 复用 YOLOv8 检测车辆（car/bus/truck），输出车辆框与外轮廓（YOLOv8-det 或 YOLOv8-seg）。
- *验收*：在测试集上车辆检出率满足指标（见 §9.4）。

**F3 — 车辆朝向识别（正/反/侧三类）**
- 对每辆检测到的车，判断朝向：正面（车头）/ 反面（车尾）/ 侧面，共 3 类；首期不区分左右侧与斜角。
- 实现方式：YOLOv8n-cls 分类（ultralytics），无尾灯先验（详见 [朝向识别-技术设计.md](朝向识别-技术设计.md)）。
- 作为分流枢纽：车尾走尾灯库检索，正/侧/不确定走整车库兜底。
- *验收*：整体准确率 ≥ 90%，反面召回率 ≥ 90%。

**F4 — 车辆部件识别（部件级）**
- 识别尾灯/前灯/后视镜/车窗/轮胎/车牌/格栅/保险杠等部件，输出部件框。
- 用途：① 辅助朝向判断（检测尾灯=车尾）；② 缩小尾灯搜索范围；③ 独立部件识别能力。
- 实现：训练 YOLOv8 部件检测器（类别=部件），详见后续部件识别技术设计。
- *验收*：部件 mAP@0.5 达标，车尾/尾灯定位准确。

**F5 — 尾灯定位与裁剪**
- 用部件检测器（runs/detect/parts_v2）在车辆内定位尾灯框（复用 `detect_taillights`，taillight mAP50=0.99，替代旧 HSV 红色分割）。
- 部件检测器直接输出尾灯框，无需左右对称配对验证。
- 裁剪尾灯区域并归一化至 224×224（复用 `crop_and_normalize`）。
- *验收*：能正确定位尾灯，灭灯也鲁棒（部件检测器不依赖颜色）。

**F6 — 亮灯/灭灯状态判定**
- 基于尾灯裁剪图 V 通道均值（亮度）判定亮/灭状态（复用 `_judge_lit`，阈值 150）。
- 亮灯与灭灯样本均入库（状态记入元数据）。
- *验收*：状态判定与人工标注一致率满足指标。

**F7 — 尾灯特征提取与入尾灯库**
- 用 ResNet-50 + ArcFace 训练的尾灯嵌入网络输出 512 维向量（详见 [特征提取网络-技术设计.md](特征提取网络-技术设计.md)）。
- 连同车型标签、朝向、亮/灭状态、样本路径写入尾灯特征库。
- *验收*：特征向量维度正确，与元数据一一对应。

**F8 — 整车特征提取与入整车库（兜底）**
- 对正面/侧面视图，用整车嵌入网络（同 ResNet-50 架构分别训练）提取 512 维向量（详见 [特征提取网络-技术设计.md](特征提取网络-技术设计.md)）。
- 写入整车特征库，标注朝向，供识别程序兜底检索。
- *验收*：整车库覆盖各车型在正/侧视图下的样本。

**F9 — 元数据与索引管理**
- 构建 FAISS 索引（尾灯库、整车库各一份）。
- 元数据 schema 见 §7.4。
- *验收*：索引可加载、可检索，元数据可查。

**F10 — 库维护**
- 支持增量入库、去重（同图/近重复样本检测）、更新、导出。
- *验收*：重复样本不重复入库，支持导出整个库。

### 5.2 识别程序（Subsystem B）

**F11 — 图像导入**
- 输入单张图片，支持单图多车。
- *验收*：正确读取并预处理。

**F12 ~ F16 — 前置流程（同建库 F2~F6）**
- 车辆检测 → 朝向识别 → 部位识别 → 尾灯定位 → 亮/灭灯判定。
- 复用建库程序的同一套前置模块，保证建库与识别特征空间一致。

**F17 — 尾灯特征提取**
- 对定位到的尾灯裁剪图提取 512 维向量（与建库同一网络）。

**F18 — 尾灯检索匹配（车尾主路径）**
- 当朝向=车尾：用尾灯特征向量在尾灯特征库中做近邻检索（FAISS）。
- 返回 Top-K 候选车型及相似度。
- *验收*：Recall@K 达标（见 §9.4）。

**F19 — 整车特征兜底判别（正/侧路径）**
- 当朝向=正面/侧面：提取整车特征，在整车特征库中检索。
- *验收*：正/侧视图下能给出合理 Top-K 候选。

**F20 — 结果融合与置信度计算**
- 按朝向分流：车尾走 F18，正/侧走 F19。
- 若车尾但尾灯检索置信度过低（如尾灯被遮挡/灭灯且库内无灭灯样本），回退到整车特征兜底。
- 输出最终车型系列 + 置信度 + Top-K。
- *验收*：融合逻辑在边界场景下行为合理。

**F21 — 结果输出与可视化**
- 在原图上标注：车辆框、朝向、尾灯框、亮/灭状态、判别车型 + 置信度。
- 保存结果图与结构化结果（JSON）。
- *验收*：可视化清晰，结构化结果可被程序解析。

## 6. 数据需求

### 6.1 数据来源
- 静态图：汽车之家 / 懂车帝（已有 [AutoInfoScraper/](AutoInfoScraper/) 爬虫模块 + chromedriver）。
- 视频抽帧：YouTube / B 站（后续扩展）。
- 自采集：停车场实拍（补充亮灯/灭灯、多角度样本）。

### 6.2 数据规模（建议）
- 车型系列数：50–100 个（首期可先 10–20 个验证流程）。
- 每车型样本：20–30 张车尾 + 5–10 张正/侧（车尾为主，正/侧用于朝向训练+整车兜底），含亮灯+灭灯。

### 6.3 标注规范
- 每张图标注：车型系列标签、朝向（正/反/侧）、亮/灭状态、**部件框**（尾灯/前灯/后视镜/车窗/轮胎/车牌等）；车型+朝向由目录名预填，部件框人工标。
- 训练集 / 测试集划分：按车型分层，约 8:2，测试集不参与建库。

### 6.4 数据与元数据存储结构

```
data/
├── raw/                  # 原始图像（raw/{车型}/，朝向子目录可选；未分朝向由标注工具自动判断+人工标）
├── processed/            # 裁剪归一化后的尾灯/整车图
├── annotations/          # 标注文件（CSV/JSON）
└── features/             # 特征库
    ├── taillight_index.faiss      # 尾灯特征索引
    ├── vehicle_index.faiss        # 整车特征索引
    ├── taillight_meta.json        # 尾灯库元数据
    └── vehicle_meta.json          # 整车库元数据
```

**尾灯库元数据 schema（每条记录）：**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int | 记录主键 |
| model_series | str | 车型系列标签 |
| brand | str | 品牌 |
| view | str | 朝向（rear/front/side） |
| lit_status | bool | 亮灯/灭灯 |
| feature | float[512] | 嵌入向量（亦存入 FAISS） |
| sample_path | str | 原始样本路径 |
| taillight_box | [x1,y1,x2,y2] | 尾灯在原图坐标 |

**整车库元数据 schema：** id / model_series / brand / view / feature[512] / sample_path / vehicle_box。

## 7. 特征库与检索设计

### 7.1 尾灯特征库（主路径）
- 嵌入网络：ResNet-50（ImageNet 预训练，去分类头），输出 512 维（详见 [特征提取网络-技术设计.md](特征提取网络-技术设计.md)）。
- 库组织：**统一库**，亮灯/灭灯样本共存，检索时按 lit_status 过滤（见 §7.3）。
- 训练：ArcFace 代理式度量损失（margin=0.5, scale=64），使同车型尾灯向量靠近、异车型远离。
- 检索：cosine 相似度（L2 归一化后内积），返回 Top-K。

### 7.2 整车特征库（兜底）
- 同 ResNet-50 架构，单独训练整车嵌入网络（输入域不同，不与尾灯网络共享权重，详见 [特征提取网络-技术设计.md](特征提取网络-技术设计.md)）。
- 用于正面、侧面视图及尾灯检索失败时的回退。

### 7.3 检索策略
- Top-K（K=1/5/10）候选 + 相似度归一化为置信度。
- 状态过滤：检索尾灯库时按查询图 lit_status 过滤（亮灯查亮灯样本，灭灯查灭灯样本），无匹配状态时回退全库。
- 车尾视图优先尾灯库；置信度低于阈值则回退整车库。

### 8.1 性能（本地 CPU 参考）
- 单图端到端识别 < 1–2 秒（不含模型首次加载）。

## 11. 风险与假设

### 11.1 假设
- 部件检测器对尾灯的检测精度足够（taillight mAP50=0.99，已验证）。
- 度量学习所需每车型样本量充足时，Recall@K 可达标。

### 11.2 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 同车型不同年款尾灯相近，按系列判别仍可能混淆 | 精度下降 | 粒度已定为"系列"以缓解；必要时引入整车特征辅助 |
| 灭灯尾灯不发光 | 亮度判定偏低 | 部件检测器不依赖颜色（taillight recall 0.967，灭灯仍可定位）；亮度阈值兜底 |
| 正/侧视图整车特征区分度有限 | 兜底路径精度低 | 明确兜底定位，降低正/侧视图的预期精度 |
| 数据量不足 | 度量学习不收敛 | 先小规模（10–20 车型）跑通流程，数据增强（albumentations） |

## 13. 术语表

| 术语 | 含义 |
|---|---|
| 尾灯 / 后车灯 | 车辆后部红色灯具 |
| 朝向 | 车辆相对镜头的方向：正面(车头)/反面(车尾)/侧面 |
| 度量学习 | 通过 Triplet Loss 等学习样本间相似度的表征 |
| Recall@K | 检索结果前 K 项中含正确答案的比例 |
| FAISS | 高维向量近邻检索库 |
| 车型系列 | 不细分年款的车型层级，如「卡罗拉」 |
